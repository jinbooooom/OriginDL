# 第一部分：架构总览与设计理念

## 第1章：OriginDL 架构概览

### 1.1 整体架构设计

OriginDL 采用分层架构设计，将系统划分为多个清晰的层次，每一层都有明确的职责，通过抽象接口实现层间解耦。这种设计使得系统具有良好的可维护性、可扩展性和可测试性。

#### 1.1.1 分层架构原理

**架构分层原理：**

分层架构的核心思想是**职责分离**和**依赖倒置**。每一层只依赖下层提供的接口，不依赖具体实现，从而实现解耦。

**四层架构设计：**

```
┌─────────────────────────────────────────────────────────┐
│  用户代码层 (User Code)                                  │
│  Tensor y = x0 + x1;                                     │
└─────────────────────────────────────────────────────────┘
                    ↓ 调用
┌─────────────────────────────────────────────────────────┐
│  第1层：Tensor (用户接口层)                              │
│  - 值语义包装                                            │
│  - 运算符重载                                            │
│  - 隐藏实现细节                                          │
└─────────────────────────────────────────────────────────┘
                    ↓ 只调用 TensorImpl 方法
┌─────────────────────────────────────────────────────────┐
│  第2层：TensorImpl (核心实现层)                          │
│  - 数据管理 (data_, grad_)                              │
│  - 计算图管理 (creator_, generation_)                   │
│  - 自动求导 (backward())                                │
└─────────────────────────────────────────────────────────┘
                    ↓ 只调用 Mat 接口方法
┌─────────────────────────────────────────────────────────┐
│  第3层：Mat (抽象接口层)                                 │
│  - 统一矩阵计算接口                                      │
│  - 多态机制                                              │
│  - 后端隔离                                              │
└─────────────────────────────────────────────────────────┘
                    ↓ 具体实现
┌─────────────────────────────────────────────────────────┐
│  第4层：后端实现层                                       │
│  ┌──────────────┐  ┌──────────────┐                     │
│  │  OriginMat   │  │  TorchMat    │                     │
│  │  (CPU/CUDA)  │  │  (LibTorch)  │                     │
│  └──────────────┘  └──────────────┘                     │
└─────────────────────────────────────────────────────────┘
```

**架构图（类图）：**

```
┌─────────────┐
│   Tensor    │  (用户接口层)
│─────────────│
│ +impl_      │──┐
│             │  │ shared_ptr
└─────────────┘  │
                 │
                 ▼
┌─────────────────────────┐
│      TensorImpl         │  (核心实现层)
│─────────────────────────│
│ +data_: shared_ptr<Mat> │──┐
│ +grad_: shared_ptr<Mat> │  │
│ +creator_: FunctionPtr  │  │
│ +generation_: int       │  │
│                         │  │
│ +backward()             │  │
└─────────────────────────┘  │
                             │
                             ▼
                    ┌─────────────────┐
                    │      Mat        │  (抽象接口层)
                    │─────────────────│
                    │ +add() = 0      │
                    │ +mul() = 0      │
                    │ +matmul() = 0   │
                    │ +clone() = 0    │
                    └─────────────────┘
                             ▲
                             │ 继承
                ┌────────────┴────────────┐
                │                         │
    ┌───────────────┐         ┌───────────────┐
    │  OriginMat    │         │  TorchMat     │
    │───────────────│         │───────────────│
    │ +storage_     │         │ +torch_tensor_│
    │               │         │               │
    │ +add()        │         │ +add()        │
    │ +mul()        │         │ +mul()        │
    └───────────────┘         └───────────────┘
```

#### 1.1.2 各层职责详解

**第1层：Tensor（用户接口层）**

**设计原理：**
- **值语义设计**：让 Tensor 对象可以像普通值一样传递，符合直觉
- **轻量级包装**：Tensor 对象只包含一个 `shared_ptr<TensorImpl>`，开销极小
- **不可变接口**：大部分操作返回新的 Tensor，不修改原对象

**类设计：**
```cpp
class Tensor {
private:
    TensorImplPtr impl_;  // 唯一的成员：shared_ptr<TensorImpl>
    
public:
    // 值语义：浅拷贝
    Tensor(const Tensor& other) : impl_(other.impl_) {}
    
    // 运算符重载（返回新对象）
    Tensor operator+(const Tensor& other) const;
    Tensor operator*(const Tensor& other) const;
    
    // 工厂方法
    static Tensor zeros(const Shape& shape);
    static Tensor ones(const Shape& shape);
    static Tensor randn(const Shape& shape);
};
```

**内存布局：**
```
Tensor 对象（栈上，8-16 字节）
  └─> impl_ (shared_ptr<TensorImpl>)
        └─> TensorImpl 对象（堆上）
              ├─> data_ (shared_ptr<Mat>)
              ├─> grad_ (shared_ptr<Mat>)
              ├─> creator_ (shared_ptr<Operator>)
              └─> generation_ (int)
```

**第2层：TensorImpl（核心实现层）**

**设计原理：**
- **数据与计算图分离**：data_/grad_ 存储数据，creator_/generation_ 管理计算图
- **智能指针管理**：使用 shared_ptr 自动管理生命周期
- **自动求导核心**：backward() 方法实现反向传播

**类设计：**
```cpp
class TensorImpl {
public:
    std::shared_ptr<Mat> data_;      // 数据（通过 Mat 抽象层）
    std::shared_ptr<Mat> grad_;       // 梯度（通过 Mat 抽象层）
    FunctionPtr creator_;              // 创建者（Operator）
    int generation_;                  // 拓扑排序用
    
    void backward();                  // 反向传播入口
    void set_creator(const FunctionPtr& func);
    void clear_grad();
};
```

**数据流关系：**
```
TensorImpl
  ├─> data_ ──> Mat ──> Storage ──> void* (实际数据)
  ├─> grad_ ──> Mat ──> Storage ──> void* (梯度数据)
  └─> creator_ ──> Operator ──> 计算图连接
```

**第3层：Mat（抽象接口层）**

**设计原理：**
- **接口与实现分离**：Mat 只定义接口，不包含实现
- **多态机制**：通过虚函数实现运行时多态
- **类型擦除**：通过基类指针隐藏具体后端类型

**接口设计：**
```cpp
class Mat {
public:
    // 纯虚函数：强制子类实现
    virtual std::unique_ptr<Mat> add(const Mat& other) const = 0;
    virtual std::unique_ptr<Mat> mul(const Mat& other) const = 0;
    virtual std::unique_ptr<Mat> matmul(const Mat& other) const = 0;
    virtual std::unique_ptr<Mat> clone() const = 0;
    
    // 属性访问
    virtual Shape shape() const = 0;
    virtual DataType dtype() const = 0;
    virtual Device device() const = 0;
    
    virtual ~Mat() = default;
};
```

**多态调用机制：**
```
TensorImpl::data_ (shared_ptr<Mat>)
  │
  ├─> 可能指向 OriginMat
  │     └─> OriginMat::add() 实现
  │
  └─> 可能指向 TorchMat
        └─> TorchMat::add() 实现

运行时根据实际类型调用对应方法
```

**第4层：后端实现层**

**OriginMat（自研后端）：**
- 从零实现，代码可读性强
- CPU 实现用于快速验证
- CUDA 实现有一定优化

**TorchMat（LibTorch 后端）：**
- 基于 PyTorch C++ 后端
- 高度优化
- 适合生产环境

#### 1.1.3 职责分离原则

**核心原则：每一层只负责自己的职责，不越界调用**

**依赖关系图：**

```
┌──────────┐
│  Tensor  │  ──┐
└──────────┘    │ 只调用 TensorImpl
                │
┌──────────────┐│
│ TensorImpl   ││  ──┐
└──────────────┘│    │ 只调用 Mat 接口
                │    │
┌──────────────┐│    │
│     Mat      ││    │  ──┐
│  (接口层)    ││    │    │ 定义接口规范
└──────────────┘│    │    │
                │    │    │
┌──────────────┐│    │    │
│  OriginMat   ││    │    │  实现 Mat 接口
│  TorchMat    ││    │    │
└──────────────┘│    │    │
                │    │    │
                └────┴────┘
```

**禁止的调用：**
- ❌ Tensor 直接调用 Mat 方法
- ❌ TensorImpl 直接调用 OriginMat/TorchMat 方法
- ❌ 算子层依赖具体后端

**允许的调用：**
- ✅ Tensor → TensorImpl
- ✅ TensorImpl → Mat（接口）
- ✅ 算子 → Mat（接口）

**设计优势：**

1. **可维护性**
   - 各层独立，修改一层不影响其他层
   - 清晰的边界，易于定位问题

2. **可扩展性**
   - 新增后端只需实现 Mat 接口
   - 算子层代码无需修改

3. **可测试性**
   - 每层可以独立测试
   - 可以 Mock 下层接口

4. **后端解耦**
   - 算子层和 Tensor 层不依赖具体后端
   - 编译时或运行时可以选择后端

### 1.2 核心设计理念

OriginDL 的设计遵循以下核心理念，这些理念贯穿整个系统的设计决策。

#### 1.2.1 值语义 vs 引用语义的选择

**设计选择：值语义**

**原理说明：**

值语义是指对象的行为像普通值一样，拷贝时创建独立副本（或共享底层数据）。引用语义是指对象的行为像引用，拷贝时共享同一份数据。

**OriginDL 的选择：值语义 + 底层共享**

**架构设计图：**

```
用户视角（值语义）：
┌──────────┐      ┌──────────┐
│ Tensor x │      │ Tensor y │  ← 两个独立的 Tensor 对象
└──────────┘      └──────────┘

底层实现（共享）：
┌──────────┐
│ Tensor x │
│──────────│
│ impl_ ───┼──┐
└──────────┘  │ shared_ptr
              │ (ref_count=2)
┌──────────┐  │
│ Tensor y │  │
│──────────│  │
│ impl_ ───┼──┘
└──────────┘
              │
              ▼
        ┌─────────────┐
        │ TensorImpl  │  ← 共享同一个 TensorImpl
        │ (ref_count) │
        └─────────────┘
```

**值语义 vs 引用语义对比：**

```
值语义（OriginDL 采用）：
  Tensor y = x;     → 浅拷贝，共享底层实现
  y = x + 1;        → 创建新 Tensor，不修改 x
  ✅ 直观、安全、易用

引用语义（不采用）：
  Tensor y = x;     → 可能共享或拷贝，不明确
  y.add(x);         → 可能修改 x，需要理解所有权
  ❌ 容易出错，需要小心处理
```

**为什么选择值语义？**

1. **易用性**：符合直觉，降低学习成本
2. **安全性**：避免意外的副作用
3. **与 PyTorch 一致**：降低迁移成本

**引用计数生命周期图：**

```
时间线：
  t0: 创建 Tensor x
      └─> TensorImpl (ref_count=1)
      
  t1: 拷贝 Tensor y = x
      └─> TensorImpl (ref_count=2)
      
  t2: x 超出作用域
      └─> TensorImpl (ref_count=1)
      
  t3: y 超出作用域
      └─> TensorImpl (ref_count=0)
          └─> 自动释放 TensorImpl
```

#### 1.2.2 可扩展性设计（多后端支持）

**设计目标：** 支持多种计算后端，便于未来扩展

**架构原理：**

通过**策略模式**和**工厂方法模式**实现多后端支持。Mat 接口定义策略，具体后端实现策略，工厂方法创建策略实例。

**架构图：**

```
┌─────────────────────────────────────────┐
│         算子层 (Operator)                │
│  Add::forward() {                       │
│    auto result = mat(x) + mat(y);        │
│  }                                       │
└─────────────────────────────────────────┘
                    │
                    │ 调用 Mat 接口
                    ▼
┌─────────────────────────────────────────┐
│         Mat 接口层 (策略接口)            │
│  virtual add() = 0;                     │
│  virtual mul() = 0;                     │
└─────────────────────────────────────────┘
                    ▲
                    │ 实现
        ┌───────────┴───────────┐
        │                       │
┌───────────────┐      ┌───────────────┐
│  OriginMat    │      │  TorchMat     │
│  (策略1)      │      │  (策略2)      │
│───────────────│      │───────────────│
│ +add()        │      │ +add()        │
│ +mul()        │      │ +mul()        │
└───────────────┘      └───────────────┘
```

**多后端支持架构：**

```
┌─────────────────────────────────────────┐
│         算子层 (Operator)                │
│  只依赖 Mat 接口，不依赖具体后端         │
└─────────────────────────────────────────┘
                    │
                    │ 调用 Mat 接口
                    ▼
┌─────────────────────────────────────────┐
│         Mat 接口层 (策略接口)            │
│  定义统一接口规范                        │
└─────────────────────────────────────────┘
                    ▲
                    │ 实现
        ┌───────────┴───────────┐
        │                       │
┌───────────────┐      ┌───────────────┐
│  OriginMat    │      │  TorchMat     │
│  (策略1)      │      │  (策略2)      │
│───────────────│      │───────────────│
│ +add()        │      │ +add()        │
│ +mul()        │      │ +mul()        │
└───────────────┘      └───────────────┘
        │                       │
        └───────────┬───────────┘
                    │
                    ▼
         ┌──────────────────┐
         │  编译时选择      │
         │  build.sh        │ → OriginMat
         │  build.sh torch │ → TorchMat
         └──────────────────┘
```

**扩展性设计：**

```
新增后端流程：
  1. 实现 Mat 接口
     ┌───────────────┐
     │ NewBackendMat │  ← 只需实现 Mat 接口
     │ +add()        │
     │ +mul()        │
     └───────────────┘
     
  2. 算子层无需修改
     ┌──────────────┐
     │   Operator   │  ← 代码不变
     └──────────────┘
     
  3. 用户代码无需修改
     ┌──────────────┐
     │  User Code   │  ← 接口保持一致
     └──────────────┘
```

#### 1.2.3 性能与易用性的权衡

**设计原则：** 在保证易用性的前提下，尽可能优化性能

**权衡策略：**

| 场景 | 易用性优先 | 性能优化 |
|------|-----------|---------|
| API 设计 | 值语义、运算符重载 | 底层共享、引用计数 |
| 内存管理 | 智能指针自动管理 | 零拷贝视图、原地操作 |
| 自动求导 | 用户只需 backward() | 计算图自动清理 |

**易用性优先的场景：**

1. **API 设计**
   ```cpp
   // 直观的运算符重载
   auto z = x + y;
   auto w = x * y;
   ```

2. **自动内存管理**
   ```cpp
   // 使用智能指针，无需手动管理
   auto x = Tensor::zeros({2, 3});
   // x 超出作用域时自动释放
   ```

3. **自动求导**
   ```cpp
   // 用户只需调用 backward()
   loss.backward();
   // 系统自动构建计算图、计算梯度、清理内存
   ```

**性能优化的场景：**

1. **零拷贝视图**
   ```cpp
   // 连续张量的 reshape 创建视图（零拷贝）
   auto y = x.reshape({3, 2});  // 共享 Storage
   ```

2. **原地操作**
   ```cpp
   // 梯度累加使用原地操作（避免内存分配）
   grad_->add_inplace(*other_grad_);
   ```

3. **计算图清理**
   ```cpp
   // backward() 后自动清理，释放内存
   loss.backward();
   // 计算图自动清理，中间结果释放
   ```

**性能与易用性权衡架构：**

```
用户层（易用性优先）：
  auto z = x + y;  ← 值语义，直观

底层实现（性能优化）：
  ┌─────────────────────────────────────┐
  │ 1. 浅拷贝共享                        │
  │    Tensor x ──┐                     │
  │               ├─> 共享 TensorImpl  │
  │    Tensor y ──┘                     │
  │                                     │
  │ 2. 算子返回 unique_ptr              │
  │    └─> 避免不必要的拷贝             │
  │                                     │
  │ 3. 转换为 shared_ptr                │
  │    └─> 开始共享管理                 │
  │                                     │
  │ 4. 多个 Tensor 共享 Storage         │
  │    └─> 零拷贝视图                   │
  └─────────────────────────────────────┘
```

#### 1.2.4 教育性与实用性平衡

**教育性：**
- OriginMat 后端：从零实现，代码可读性强
- 清晰的架构分层：便于理解深度学习框架原理
- 完整的注释：解释设计决策和实现细节

**实用性：**
- 支持 LibTorch 后端：生产环境可用
- 性能优化：零拷贝、原地操作等
- 完整功能：自动求导、神经网络模块等

```
  │     │                               │
  │     ├─> inputs_ (vector<Tensor>)   │
  │     └─> outputs_ (weak_ptr)        │
  │                                     │
  └─────────────────────────────────────┘
```

#### 1.3.3 Mat 层（抽象接口层）

**类图（继承关系）：**

```
                    ┌─────────────┐
                    │     Mat     │  (抽象基类)
                    │─────────────│
                    │ +add() = 0  │
                    │ +mul() = 0  │
                    │ +matmul()=0 │
                    │ +clone() = 0│
                    └─────────────┘
                           ▲
                           │ 继承
            ┌──────────────┴──────────────┐
            │                             │
    ┌───────────────┐           ┌───────────────┐
    │  OriginMat    │           │  TorchMat     │
    │───────────────│           │───────────────│
    │ -storage_     │           │ -torch_tensor_│
    │               │           │               │
    │ +add()        │           │ +add()        │
    │ +mul()        │           │ +mul()        │
    │ +matmul()     │           │ +matmul()     │
    └───────────────┘           └───────────────┘
```

**多态调用架构：**

```
多态调用流程：
  ┌─────────────────────────────────────┐
  │ TensorImpl::data_->add(*other)     │
  └─────────────────────────────────────┘
              │
              │ 运行时多态
              ▼
    ┌─────────────────────┐
    │  虚函数表查找       │
    └─────────────────────┘
              │
      ┌───────┴───────┐
      │               │
      ▼               ▼
┌──────────┐    ┌──────────┐
│OriginMat │    │TorchMat  │
│::add()   │    │::add()   │
└──────────┘    └──────────┘
```

#### 1.3.4 后端实现层

**后端选择机制：**

```
编译时选择：
  build.sh          → OriginMat (默认)
  build.sh torch    → TorchMat

运行时使用：
  TensorImpl::data_ 可以指向任意 Mat 子类
  通过多态调用对应实现
```

### 1.4 数据流路径分析

#### 1.4.1 前向传播路径

**完整调用链：**

```
┌─────────────────────────────────────────────────────┐
│ 用户代码层                                           │
│ Tensor y = x0 + x1;                                 │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ Tensor 层                                            │
│ operator+(x0, x1)                                    │
│   └─> 创建 Add 算子，调用 operator()                 │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ Operator 层                                          │
│ Add::operator()(inputs)                              │
│   ├─> forward(inputs)                               │
│   ├─> 设置 creator_                                  │
│   └─> setup_computation_graph()                      │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ Add::forward()                                       │
│   └─> mat(xs[0]) + mat(xs[1])                       │
│         └─> 调用 Mat 接口                            │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ Mat 接口层（多态调用）                               │
│ OriginMat::operator+() 或 TorchMat::operator+()      │
│   └─> 返回 unique_ptr<Mat>                           │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 转换层                                               │
│ convert_mat_to_tensor(std::move(result))             │
│   └─> Tensor(std::unique_ptr<Mat>)                   │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ TensorImpl 层                                        │
│ TensorImpl(std::unique_ptr<Mat>)                     │
│   └─> unique_ptr → shared_ptr 转换                   │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 后端层                                               │
│ OriginMat 持有 Storage                               │
│   └─> Storage 持有实际数据 (void*)                   │
└─────────────────────────────────────────────────────┘
```

**关键转换点：**

1. **Tensor → Mat**：通过 `impl_->data_` 获取 Mat 引用
2. **Mat → unique_ptr<Mat>**：算子返回计算结果
3. **unique_ptr → shared_ptr**：在 TensorImpl 构造函数中转换
4. **所有权转移**：从算子层转移到 TensorImpl 层

#### 1.4.2 反向传播路径

**完整调用链：**

```
┌─────────────────────────────────────────────────────┐
│ 用户代码层                                           │
│ loss.backward();                                     │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ TensorImpl::backward()                               │
│   1. 初始化梯度（全1）                                │
│   2. 拓扑排序                                        │
│   3. 遍历计算图                                      │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 拓扑排序                                             │
│   按 generation_ 从大到小排序                        │
│   使用 list + set 维护队列                           │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 遍历计算图                                           │
│   for each Operator:                                 │
│     1. 收集输出梯度 (weak_ptr → shared_ptr)         │
│     2. 调用 Operator::backward(gys)                  │
│     3. 获取输入梯度 gxs                              │
│     4. 梯度累积                                      │
│     5. 递归处理输入的 creator_                       │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 梯度累积                                             │
│   if (grad_ == nullptr)                             │
│       grad_ = gx.impl_->data_;  // 直接共享         │
│   else                                               │
│       grad_->add_inplace(*gx.impl_->data_);  // 累加│
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│ 计算图清理                                           │
│   1. 清理 inputs_                                    │
│   2. 清理中间 tensor 的 grad_ 和 creator_            │
│   3. 断开循环引用                                    │
└─────────────────────────────────────────────────────┘
```

**拓扑排序原理：**

```
计算图结构：
  x (generation=0)
    ↓
  Add (generation=1)
    ↓
  y (generation=1)
    ↓
  Mul (generation=2)
    ↓
  z (generation=2)

拓扑排序结果（从大到小）：
  [Mul, Add]  → 执行顺序：Mul → Add
```

#### 1.4.3 视图操作路径

**视图创建原理：**

```
连续张量的 reshape：
  ┌─────────────────┐
  │  Storage       │  (数据拥有者)
  │  [1,2,3,4,5,6] │
  └─────────────────┘
         ▲
         │ 共享
    ┌────┴────┐
    │         │
┌───────┐ ┌───────┐
│ Mat1  │ │ Mat2  │  (视图)
│(2,3)  │ │(3,2)  │
└───────┘ └───────┘
    │         │
    └─────────┘
    共享 Storage，零拷贝
```

**实现流程：**

```
Tensor::reshape(shape)
  ↓
TensorImpl::reshape(shape)
  ↓
data_->reshape(shape)
  ↓
判断 is_contiguous()
  ├─> true: 创建视图 (view)
  │     └─> 共享 Storage，零拷贝
  │
  └─> false: 创建新 Storage
        └─> 拷贝数据
```

### 1.5 动态图 vs 静态图

#### 1.5.1 动态图（Dynamic Graph）

**原理：**

动态图在**运行时**构建计算图。每次前向传播时，算子调用会记录输入输出关系，形成计算图。

**计算图构建过程：**

```
前向传播时自动构建：

x (generation=0)
  ↓
Add::forward({x, 1})
  ├─> 计算 y = x + 1
  ├─> 设置 y.creator_ = Add
  └─> 设置 y.generation_ = 1
  ↓
y (generation=1)
  ↓
Mul::forward({y, 2})
  ├─> 计算 z = y * 2
  ├─> 设置 z.creator_ = Mul
  └─> 设置 z.generation_ = 2
  ↓
z (generation=2)
```

**计算图结构：**

```
节点：Tensor
  ├─> data_: 数据
  ├─> grad_: 梯度
  ├─> creator_: 创建者（Operator）
  └─> generation_: 代数

边：Operator
  ├─> inputs_: 输入 Tensor
  └─> outputs_: 输出 Tensor (weak_ptr)

计算图：
  x ──[Add]──> y ──[Mul]──> z
```

**优势：**
- **灵活性**：可以根据输入动态改变计算图
- **易调试**：可以打印中间结果
- **易用性**：用户代码直观

**劣势：**
- **性能开销**：每次前向传播都要构建计算图
- **优化受限**：难以进行全局优化

#### 1.5.2 静态图（Static Graph）

**原理：**

静态图在**编译时或加载时**构建计算图。图结构固定，执行时按预定义的顺序运行。

**计算图构建过程：**

```
加载模型文件：
  PNNXGraph graph("model.param", "model.bin");
  graph.build();
    ├─> 解析 .param 文件（图结构）
    ├─> 加载 .bin 文件（权重）
    ├─> 创建节点和边
    ├─> 拓扑排序
    └─> 创建 Operator

执行：
  graph.forward();
    └─> 按拓扑序执行算子
```

**计算图结构：**

```
节点：PNNXNode
  ├─> type: 算子类型
  ├─> name: 节点名称
  ├─> input_names: 输入名称
  ├─> output_names: 输出名称
  ├─> op: Operator 实例
  └─> execution_order: 执行顺序

边：通过 input_names 和 output_names 建立

计算图（固定）：
  input ──[Conv2d]──> conv1 ──[ReLU]──> relu1 ──[Linear]──> output
```

**优势：**
- **性能优化**：可以预先优化计算图
- **内存优化**：可以预先分配内存
- **推理效率**：适合部署场景

**劣势：**
- **灵活性差**：计算图结构固定
- **调试困难**：难以查看中间结果

#### 1.5.3 两种模式的适用场景

**动态图适用于：**
- 训练阶段：需要灵活调整模型结构
- 研究阶段：需要快速实验
- 调试阶段：需要查看中间结果

**静态图适用于：**
- 推理阶段：性能要求高
- 部署场景：模型结构固定
- 生产环境：需要稳定性和性能

**OriginDL 的支持：**
- **动态图**：默认模式，支持自动求导
- **静态图**：通过 PNNX 支持，用于推理部署

## 第2章：项目结构与编译系统

### 2.1 目录结构设计

#### 2.1.1 模块划分原则

**模块化设计原理：**

模块化设计将系统划分为多个独立的模块，每个模块有明确的职责和接口。模块之间通过接口交互，降低耦合度。

**目录结构：**

```
origindl/
├── include/              # 头文件（接口定义）
│   └── origin/
│       ├── core/         # 核心模块
│       │   ├── tensor.h
│       │   ├── tensor_impl.h
│       │   ├── operator.h
│       │   └── parameter.h
│       ├── mat/          # 矩阵计算模块
│       │   ├── mat.h     # Mat 接口
│       │   ├── origin/  # OriginMat 实现
│       │   └── torch/   # TorchMat 实现
│       ├── operators/    # 算子实现
│       ├── nn/           # 神经网络模块
│       ├── optim/        # 优化器模块
│       └── pnnx/         # 静态图模块
├── src/                  # 源文件（实现）
│   └── (与 include 对应)
├── tests/                # 测试代码
│   ├── unit_test/        # 单元测试
│   ├── example/          # 示例代码
│   └── compare/          # 对比测试
├── docs/                 # 文档
├── 3rd/                  # 第三方依赖
└── build.sh              # 编译脚本
```

**模块依赖关系：**

```
┌─────────┐
│  core   │  (核心模块，不依赖其他模块)
└─────────┘
     │
     ├─> mat (矩阵计算模块)
     │
     ├─> operators (算子模块)
     │     └─> 依赖 core, mat
     │
     ├─> nn (神经网络模块)
     │     └─> 依赖 core, operators
     │
     ├─> optim (优化器模块)
     │     └─> 依赖 core, nn
     │
     └─> pnnx (静态图模块)
           └─> 依赖 core, operators
```

#### 2.1.2 头文件组织

**组织原则：**
- **include/ 与 src/ 分离**：清晰的接口定义
- **命名空间**：使用 `origin` 命名空间
- **模块化**：每个模块独立的目录

**示例：**
```
include/origin/core/tensor.h     → src/core/tensor.cpp
include/origin/mat/mat.h         → src/mat/mat.cpp
include/origin/operators/add.h   → src/operators/add.cpp
```

#### 2.1.3 测试代码组织

**三种测试类型：**

1. **unit_test/** - 单元测试
   - 测试单个函数或类
   - 使用 GoogleTest 框架

2. **example/** - 示例代码
   - 完整的使用示例
   - 展示实际应用场景

3. **compare/** - 对比测试
   - 与 PyTorch 数值对比
   - 验证实现正确性

```cmake
# C++ 标准
```
